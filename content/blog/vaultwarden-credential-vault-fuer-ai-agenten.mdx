AI-Agenten brauchen Credentials — API-Keys, Passwörter, Tokens. Die Frage ist: Wie verwaltet man diese Secrets sicher, wenn nicht ein Mensch sie nutzt, sondern ein autonomer Software-Agent?

In diesem Beitrag beschreiben wir, wie wir **Vaultwarden** als zentralen Credential-Vault für unsere Paperclip AI-Agenten einsetzen.

## Das Problem: Secrets in einer Agenten-Welt

Ein klassischer Mensch öffnet seinen Passwort-Manager, kopiert ein Passwort und schließt ihn wieder. Ein AI-Agent funktioniert anders:

- Er braucht Credentials **programmatisch**, ohne GUI-Interaktion
- Er läuft **autonom** — nachts, als Hintergrundprozess, ausgelöst durch Events
- Er darf Secrets **nicht im Klartext** in Config-Dateien oder Code haben
- Seine **Zugriffe sollen nachvollziehbar** sein (Audit-Log)

Die klassische Lösung — Secrets in `.env`-Dateien oder Docker-Secrets — löst das Problem nur teilweise. Zentrales Credential-Management mit Audit-Trail ist damit nicht möglich.

## Warum Vaultwarden?

**Vaultwarden** ist eine inoffizielle, ressourcenschonende Reimplementierung des Bitwarden-Servers in Rust. Es ist mit allen Bitwarden-Clients kompatibel und bietet gegenüber dem offiziellen Bitwarden-Server einige Vorteile für Self-hosting:

| Feature | Bitwarden (offiziell) | Vaultwarden |
|---|---|---|
| Self-hosted | Ja (ressourcenintensiv) | Ja (minimal) |
| RAM-Bedarf | ~2 GB | ~50 MB |
| Alle Features | Ja (kostenpflichtig) | Ja (kostenlos) |
| Bitwarden-kompatibel | — | Vollständig |

Für unser HomeLab war die Wahl klar: Vaultwarden läuft als leichtgewichtiger Docker-Container und bietet dennoch alle Features, die wir brauchen.

## Setup: Vaultwarden als Docker-Stack

Unser Vaultwarden-Stack läuft auf einem dedizierten Docker-Node hinter Traefik:

```yaml
services:
  vaultwarden:
    image: vaultwarden/server:latest
    container_name: vaultwarden
    restart: unless-stopped
    environment:
      DOMAIN: "https://vault.strausmann.cloud"
      SIGNUPS_ALLOWED: "false"
      ADMIN_TOKEN: "${ADMIN_TOKEN}"
    volumes:
      - ./data:/data
    labels:
      - "traefik.enable=true"
      - "traefik.http.routers.vaultwarden.rule=Host(`vault.strausmann.cloud`)"
      - "traefik.http.routers.vaultwarden.tls.certresolver=letsencrypt"
```

Wichtig: `SIGNUPS_ALLOWED: "false"` — neue Accounts können nur der Admin anlegen.

## Der Bitwarden Secrets Manager für Agenten

Bitwarden (und damit Vaultwarden) bietet neben dem klassischen Passwort-Manager auch einen **Secrets Manager** — speziell für maschinelle Zugriffe konzipiert:

- **Service Accounts** statt menschlicher Nutzer
- **Access Tokens** für programmatischen Zugriff
- **Granulare Berechtigungen** pro Secret
- **Audit-Log** für jeden Zugriff

### Service Account anlegen

Im Vaultwarden/Bitwarden-UI:

1. **Organisation** anlegen (z.B. "HomeLab Agents")
2. **Secrets Manager** aktivieren
3. Unter **Service Accounts** → "Neuer Service Account"
4. Token generieren und sicher speichern

Jeder AI-Agent erhält einen eigenen Service Account — so ist nachvollziehbar, welcher Agent wann auf welches Secret zugegriffen hat.

## Integration mit Paperclip AI

Unsere Paperclip-Agenten nutzen den `bw serve`-Ansatz: Ein lokaler Bitwarden-CLI-Server läuft auf dem Host und stellt eine REST-API bereit, über die Agenten Secrets per MCP-Tool abrufen können.

```bash
# Bitwarden CLI entsperren und lokalen Server starten
bw unlock --passwordenv BW_MASTER_PASSWORD
bw serve --port 8087
```

Der MCP-Server (`vault-lookup`) der Agenten kommuniziert dann mit diesem lokalen Dienst:

```
GET http://localhost:8087/object/item/{secret-name}
```

### Skill: vault-lookup

Für den Zugriff aus Agenten-Konversationen haben wir einen dedizierten Sub-Agenten:

```
Agent(subagent_type="vault-lookup", model="haiku",
  prompt="Gib mir den API-Key für Hetzner Cloud")
```

Der `vault-lookup`-Agent hat ausschließlich Lesezugriff und gibt nur das angeforderte Secret zurück — keine anderen Vault-Inhalte.

## Best Practices

### 1. Least Privilege

Jeder Service Account hat nur Zugriff auf die Secrets, die er tatsächlich braucht. Der Infrastruktur-Agent hat keinen Zugriff auf M365-Credentials und umgekehrt.

### 2. Rotation-Strategie

Secrets werden in drei Kategorien eingeteilt:

| Kategorie | Rotations-Intervall | Automatisierung |
|---|---|---|
| API-Keys (extern) | 90 Tage | manuell |
| Service-Tokens (intern) | 180 Tage | geplant via n8n |
| SSH-Keys | jährlich | manuell |

### 3. Kein Plaintext in Code oder Logs

Agenten dürfen Secrets nur im Arbeitsspeicher halten, niemals:
- in Kommentare am Issue schreiben
- in Log-Dateien ausgeben
- in Konfigurations-Commits committen

In unseren Agenten-Prompts steht explizit: *"Keine Geheimnisse oder private Daten nach außen übertragen."*

### 4. Audit-Log regelmäßig prüfen

Vaultwarden schreibt jeden Zugriff ins Admin-Log. Ein wöchentlicher n8n-Workflow prüft auf ungewöhnliche Zugriffsmuster und meldet per Gotify-Notification.

## Erfahrungen nach mehreren Monaten Betrieb

Was gut funktioniert:
- **Zentrale Verwaltung** — alle Credentials an einem Ort, kein Suchen mehr in Config-Dateien
- **Transparenz** — Audit-Log zeigt genau, welcher Agent wann auf was zugegriffen hat
- **Rotation ist einfach** — Secret im Vault ändern, alle Agenten nutzen automatisch das neue

Was wir gelernt haben:
- **`bw serve` ist zustandsbehaftet** — bei Server-Neustarts muss entsperrt werden. Wir haben das durch einen systemd-Service mit automatischem Entsperren beim Start gelöst.
- **Zu viele Secrets in einer Collection** — macht Suche unübersichtlich. Wir strukturieren jetzt nach Team: `infra/`, `security/`, `engineering/`, `m365/`.

## Fazit

Vaultwarden ist für unser Szenario die perfekte Lösung: ressourcenschonend, Bitwarden-kompatibel, Self-hosted und mit dem Secrets Manager bestens für programmatische Zugriffe geeignet.

Der Schlüssel ist die Kombination: Vaultwarden als zentraler Store, `bw serve` als lokale API, dedizierter MCP-Sub-Agent für den Zugriff aus Agenten-Konversationen — und ein klares Berechtigungskonzept, das jedem Agenten nur Zugriff auf das gibt, was er wirklich braucht.
